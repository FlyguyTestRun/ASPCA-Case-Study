# ADR 0041: Corrections from a second, independent review of the deployed page

Status: accepted. Date: 2026-07-14.

## Problem

A careful, independent read of the deployed page, checking claims against the actual source rather than trusting the page's own prose, surfaced six items. Four were real defects; one was a genuine ambiguity in accurate text; one, checked directly against the code and its tests, turned out to already be correct as designed.

## Findings and decisions

**The "181 to 141" line count was real but ambiguously worded.** Checking `original/charity-donor-outreach/SKILL.md` on GitHub shows 173 lines, not 181, which reads as an 8-line overstatement. Git history shows the true story: the rewritten skill file (`skill/charity-donor-outreach/SKILL.md`) had grown to 181 lines of its own, across the whole project, before a later documentation-only pass trimmed it to 141 ([ADR 0036](0036-skill-md-trimmed-to-operational-content.md)). "181" was never about the original file; it was the rewrite's own line count immediately before that trim. The number was accurate, but every place it appeared sat next to original-vs-rewrite framing, inviting exactly the reasonable misread this review made. Reworded in the deliverable, `deliverable/README.md`, and ADR 0037 to state plainly that this is the rewrite's own before/after, not a comparison to the original.

**The walkthrough tour could contradict its own narration.** Step 6 of 7 spotlights Shirley Magnusdottir with the claim "filed as Silver, computes to Gold." If a visitor clicked "Apply suggested corrections" (now the page's most prominent button, [ADR 0039](0039-guided-process-nav-and-final-snapshot-in-archive.md)) before starting the tour, her row would already read Gold/Gold, directly contradicting the narration the moment it highlighted her. `startTour()` now resets the dataset to the canonical, uncorrected state before every run, the same reset `resetBtn` performs, so the tour always demonstrates against the state its script was actually written for regardless of what a visitor did first.

**The process nav's status filter had no reset, producing a silent dead end.** Step 2 ("Errors found") sets the status filter to "Needs correction" and nothing ever reset it. Once corrections were applied and zero records needed correction, clicking that step produced a bare "No donors match this search or filter" with no explanation, and the filter stayed stuck on every subsequent step, hiding rows a later step (like the sign-off checklist) needed to show. Every step now explicitly resets the filter to "all" except step 2, which only applies the "Needs correction" filter when there is at least one such record; otherwise it falls back to "all" and states plainly that every mismatch has already been fixed.

**"Ten defects" read as contradicting the trap registry's count.** The written assessment groups the original's problems into ten narrative themes; the trap registry itemizes them individually, thirteen rows tied to the original skill and data (numbers 1 through 13; 14 and 15 are defects found in the rewrite itself, correctly excluded from a count about the original). The grouping is real and defensible (several registry rows share one root cause, like the four tier mismatches all stemming from "stated values were never verified"), but nothing on the deployed page said so, so the two numbers looked like a contradiction sitting side by side. Reworded to state both counts and name the grouping directly: ten themes in the assessment, thirteen individual items in the registry, several sharing one theme.

**The letter date drifting away from the as_of_date** is real and is documented separately in [ADR 0031's amendment](0031-letter-date-is-not-the-as-of-date.md#why-the-committed-demo-pins-letter_date-to-the-as_of_date-too): the committed demo now pins `letter_date` to the same `as_of_date` so the two stop drifting apart on every rebuild.

**Susan Nakamura's confidence landing at exactly 0.90 was checked and found to already be correct as designed, not a gap.** `review_level()` uses a strict `confidence < REVIEW_MANDATORY_BELOW` comparison; at exactly 0.90 that is false, so her record gets `"recommended"` review, not `"mandatory"`, and `confidence_band()` (same strict comparison) puts her in the `"pass"` band, not `"report"`. This is not silent: a "recommended" pill is visible on her row, she is not treated as fully clean. It is also not accidental: [ADR 0011](0011-confidence-scoring-feedback-loop.md) states the boundary explicitly ("0.90 and above: pass. Any warning still flags the letter for recommended review"), and `tests/test_rules.py` pins both boundary values directly (`confidence_band(0.90) == "pass"`, `review_level("Silver", 0.9, []) == "recommended"`). No code changed; this entry exists so the review's specific question has a specific, evidenced answer on the record.

## What this changes going forward

Rebuilt the deliverable and re-verified live: the tour resets state correctly, the process nav's filter no longer strands a visitor on an empty table, and the wording fixes read unambiguously. Suite holds at 155, all passing.
