# ADR 0044: The walkthrough drops embedded audio and becomes a live demonstration

Status: accepted. Date: 2026-07-14.

## Problem

The guided tour built in [ADR 0021](0021-standalone-review-artifact.md) was a six-then-seven-beat spotlight-and-caption slideshow, capped at two minutes, with a synthesized narration track. Three things about it stopped being right as the deliverable grew:

- **The narration was never going to stop sounding robotic.** The embedded voice was always a placeholder, generated by the Windows speech synthesizer, meant to be swapped for a real recording. It never was, and every rebuild of the demo kept re-synthesizing the same robotic voice against whatever the script currently said, which is a problem no amount of script editing fixes.
- **The tour only ever narrated past tense.** It spotlighted a live part of the page and described what the system does, but never actually did anything: applying a correction, checking off a review, exporting an archive all happened off-screen, if at all, left for a visitor to try afterward on their own.
- **Two minutes stopped being enough room.** The deliverable had grown an architecture section, a restraint section, a scripts panel, and a merge feature since the budget was set, and none of it was in the tour. A visitor watching the two-minute version and nothing else would miss most of what the page had grown to explain.

## Decision

**Audio comes out entirely**, not replaced with a better placeholder. There is no way to make a synthesized voice sound less robotic; the actual fix is a human recording, and the project never had one to embed. Captions carry the whole tour on their own, and they read fine aloud if someone wants to record narration over a screen capture of the tour itself, using it as a de facto script.

**The tour becomes a live demonstration.** Steps can now carry a real `action`, a function that performs an actual operation on the actual embedded dataset: filtering the table to the flagged rows, applying the suggested correction, checking off every mandatory review, opening the upload panel. `showStep()` no longer mutates state forward incrementally; it resets to the canonical dataset and replays every action from step 0 through the target index on every step change, so Back, Next, and jumping to a step directly all land on exactly the state a fresh visitor would see there, regardless of navigation direction, and ending the tour early always returns the page to a clean, undemoed state.

**The tour grows to sixteen steps, uncapped**, covering four movements: what the original prompt would have done and why that put Doug's own goals (consistent, reliable, scalable) at risk; the architecture stage by stage, each one naming its real script and its decision record, including a step on `SKILL.md` itself as the file that actually orchestrates the other three; a five-step live demonstration that finds the flagged donors, applies a real correction, signs off review, and opens the export gate, watching Shirley Magnusdottir's tier flip from Silver to Gold on screen, not narrated after the fact; and a step on running a different donor file through the same checks.

**Two smaller, related additions.** "Mark all reviewed" and "Clear all" buttons apply to every mandatory-review donor at once, since the live demonstration's sign-off step needed exactly that action and a reviewer with a real file has the same need. The `dist/` delivery package now includes a copy of the revised `SKILL.md` alongside `ASSESSMENT.docx` and `donor-data-review.html`, since it is one of the three things worth reading first and belongs in the package that actually travels.

## What this changes going forward

Dropping the embedded audio also dropped the built file from 8.4 MB to 0.2 MB, since the base64-encoded narration track was nearly the entire weight of the file; a side effect of fixing the robotic-voice problem rather than the point of it. `embed_dataset.py` no longer knows anything about audio at all, not even as a fallback path.

`showStep()`'s reset-and-replay design means the two hardest bugs a live-action tour could have, forward/backward navigation producing different state, and ending mid-demo leaving the page visibly altered, are structurally impossible rather than something to remember to handle at each step. Verified live end to end: every live-action step performs its real operation (filter set, correction applied and visible on Shirley's actual row, all five mandatory donors checked off, archive gate genuinely unlocked, upload panel genuinely opened), backward navigation from the last step to a live-action step reproduces the exact same state forward navigation produced, and ending the tour from any step returns the page to the same clean state as a fresh load. A real, unrelated bug surfaced while writing the automated version of this check: the Node test stub's `getElementById` returned a fresh, disconnected element on every call instead of the same node a real browser returns for a given id, so a value written by one step's action was invisible to a later read of the same element. Fixed by caching stub elements by id, matching real DOM identity semantics; this was a test-only bug, not a page bug, but it would have made every future live-action test pass by accident regardless of whether the real page worked, the same failure mode ADR 0021 already caught once in this same stub. Six new tests exercise the actual `replayActionsThrough()` function the page uses, not a reimplementation of its logic, so "the tour actually does what it says" is enforced, not asserted. Suite grows from 157 to 160.

`tutorial_transcript.md` and the placeholder narration `.wav` are untouched; they still serve the separate Streamlit review app's own audio walkthrough toggle, a different feature this change does not touch.

## Addendum: the three pipeline stages split into three steps

The tour's stage 6 originally covered calculate and generate together, spotlighting a single invisible bridge rect (`dagNodeCalcGen`) spanning both visible DAG nodes, since the two were treated as one beat ("money and language, not guesses"). That does not match the pipeline diagram itself, on this page and in `README.md`'s own mermaid version, which draws them as two distinct stages with two distinct scripts. Asked directly to make each stage individually explainable, split the combined step into two: calculate (`dagNodeCalculate`, `calculate_ask.py`, ADR 0003) and generate (`dagNodeGenerate`, `generate_letters.py`, ADR 0017, the letter-schema-validation record rather than the arithmetic one, since that is what this stage actually guarantees). The now-unused bridge rect was removed. The tour grows from sixteen to seventeen steps; every step from the human gate onward shifts back one index accordingly. Verified live: each of the three stage nodes highlights at a distinct, correct screen position when its step is active, and the renumbered live-action steps still perform their real operations correctly, covered by the same six live-action tests, updated to the new indices.
