SKILL.md, and why there is a harness around it

What changed in SKILL.md

SKILL.md is the file an AI assistant reads to trigger this skill, not a design document; every line in it costs context at the exact moment that matters most. It has been trimmed twice, each pass cutting rationale rather than structure. The rewrite itself (the original is 173 lines) first grew to 181 as pipeline detail and decision links accumulated across the project, then came down to 141 by moving that rationale out to one-line ADR pointers (ADR 0036). A second pass then brought it to 100, net: the pipeline overview table came out (this page's own diagram above says the same thing to a human, and an agent already gets the exact commands below it), a paragraph explaining why personalization is optional, a closing list of reference documents, a redundant opening paragraph that restated three rules the Hard Rules checklist below already states, and one line went back in on a third read: the prompt-injection guardrail in step 5 had lost its actual required action, "flag the record instead," leaving only the passive principle. That clause changes what the agent does next, none of the cuts do, which is exactly the line this whole pass was drawn on does next (ADR 0042). Every command across both passes is unchanged, byte for byte, confirmed by diff each time. What remains: the required inputs, every workflow step's exact command, the personalization guardrails, and the five hard rules, because those are what an agent actually has to follow to run this correctly, not background on why.

Why there is a harness at all

A well written prompt only ever raises the odds a model behaves. It never guarantees it. That is not a criticism of prompting; it is a description of what a prompt can and cannot do. A probabilistic system asked to also be a database, a calculator, and a compliance officer will eventually get one of those jobs wrong, quietly, in a way nobody notices until a donor gets the wrong letter, which is exactly what the comparison above demonstrates on a real record.

The harness is everything built around the one place a model is allowed to touch language, converting as much of that probability into a guarantee as the problem allows:

  • Schema validation, twice. Once on the raw donor row (donor.schema.json) before any business rule runs, once on the assembled letter (letter_schema.json) before any HTML exists. ADR 0024, ADR 0017.
  • Deterministic arithmetic. Every tier, every date check, every ask amount is one fixed function, tested, never a model guess. ADR 0002, ADR 0003.
  • A confidence rubric that escalates instead of hiding. Below 0.70 blocked outright, below 0.90 held for a person, and every band emits a structured escalation event, so uncertainty is visible, not swallowed. ADR 0011.
  • Mandatory human gates on the highest-stakes records. Every Platinum donor, always, regardless of confidence. ADR 0009.
  • One bounded exception. An optional, off-by-default personalization step, walled off in its own versioned prompt file, that can only ever touch phrasing, never a fact, a number, or a claim. ADR 0025.

Bounded probability where a guarantee is not possible, a guarantee everywhere it is. That is the architecture, in one sentence.

What this deliberately does not build yet

A harness can be over-built the same way a prompt can be under-built. Knowing when not to use an LLM extends to knowing when not to add infrastructure: every item below was considered and left out on purpose, each with the specific condition that would justify adding it. The full table, with every deferred component: docs/scale-architecture.md.

  • No orchestration framework. A linear three-stage pipeline does not need a graph engine. Add one when the workflow gains real branches: multi-step approvals, parallel campaign runs, human review resumed days later.
  • No retrieval or vector database. Letter language comes from an approved library on purpose; retrieval adds variance where this needs none. Add one only if the system starts answering open questions instead of filling governed templates.
  • No per-user identity and access management. A single-reviewer demo cannot honestly implement enterprise identity; pretending would be theater. Add it when the app is deployed for a team, so every approval and adoption records a real identity.
  • No machine learning. Fifty records cannot train anything; the confidence rubric's penalty weights have no outcome data yet to learn from. Add it once a few hundred reviewer decisions exist to calibrate against, with the same fail, report, and pass gates unchanged around it.

Why the pipeline is shaped this way

The original skill asked a language model to hold the donor list, do the arithmetic, and write the letter, all in one step, on faith. The redesign puts a checkpoint at every place that step could go wrong, before a model ever touches the data.

deterministic pipeline stage where a planted error is caught

Tiers are computed, not trusted

Four donors were filed under a tier their own giving history contradicts. The gift list is the source of truth; the stated label is checked against it every run.

Dates need a reference point

One donor's gift is dated a year later than the rest of the file. Every date rule now runs against an explicit as-of date instead of the day the batch happens to run.

Letters are data before they are HTML

Each letter is checked against a schema, exactly one dollar amount, every field present, before any markup exists.

Confidence decides who reviews what

Below 0.70 a record is blocked outright. Below 0.90 it is held for a person. Everything else still gets a Platinum-always check.

Same donor, two approaches

The clearest way to answer "how are the outputs different": one real donor, run through what the original instructions literally say to do, and through what this pipeline actually does. Neither number is hypothetical; both are the real formula, applied to Ruth Andersen's real gift history (the original SKILL.md, row 24 of the donor table).

Original SKILL.md, followed literally
  1. Look up her tier in the table: Silver, as stated. The instructions never say to check it against anything.
  2. Silver asks for 15% of her largest gift: 15% × $7,000 = $1,050
  3. Round to the nearest $50 here, before the uplifts below (the instructions' own step order): $1,050
  4. She gave last year: ×1.10 loyalty uplift = $1,155
  5. She is a volunteer: +$100 flat = $1,255
  6. Emergency appeal: ×1.2 = $1,506, and nothing rounds it again
$1,506 what a model following these instructions, on this donor, would ask for
This pipeline, run on the same donor
  1. Recompute her tier from the actual gift history: $3,000 + $4,000 + $5,000 + $6,000 + $7,000 = $25,000 lifetime → Gold, not Silver
  2. Stated Silver contradicts computed Gold: held in exceptions, no letter, until a person approves the correction
  3. Corrected to Gold, recalculated: 25% × $7,000 = $1,750
  4. Loyalty uplift ×1.10 = $1,925
  5. Volunteer +$100 = $2,025
  6. Emergency appeal ×1.2 = $2,430, rounded once, at the end: $2,450
$2,450 the ask this pipeline computes, and the only one it ever sends

$944 apart, on one donor, entirely because a stated label was never checked against the money it was supposed to describe. That same mismatch, not a different one, is why four donors in this fifty-record file were held rather than mailed at the original, unverified numbers, and why the findings on this page are not the findings the original prompt would have produced.
